# ADR-001: 服务模式选择

## 状态

已接受

## 上下文

基于第一阶段需求调研，我们发现图书馆用户需求呈现多层次特征：
- **显性需求**：查书找座位等基础工具性需求
- **隐性需求**：心理支持、社群连接等情感性需求
- **潜在需求**：个性化学习路径等智能化需求

需要决定系统服务模式以满足不同层次需求。

## 决策

我们选择**陪伴型助手 + 社区型平台混合模式**，理由如下：

### 选项分析

#### 选项A：工具型系统
**定义**：系统仅提供功能性服务，如座位查找、图书检索、借还记录等

**优点**：
- 开发成本低，技术架构简单
- 用户使用目的明确，效率高
- 维护和迭代容易

**缺点**：
- 仅满足显性需求，无法触及隐性需求
- 用户粘性低，容易替代
- 无差异化竞争优势
- 用户情感连接缺失

**适用场景**：
- 用户需求明确且单一
- 技术团队规模小
- 快速上线需求

#### 选项B：陪伴型助手
**定义**：系统不仅提供功能，还提供个性化推荐、学习陪伴、情感支持等服务

**优点**：
- 满足显性需求和部分隐性需求
- 用户粘性高，使用时长增加
- 建立用户情感连接
- 提供差异化服务

**缺点**：
- 开发成本中等，需要AI能力
- 技术架构复杂度增加
- 需要持续迭代优化
- 用户期望管理困难

**适用场景**：
- 用户需求多层次
- 用户粘性需求高
- 技术团队有AI能力

#### 选项C：社区型平台
**定义**：系统不仅提供服务，还构建用户社区，促进用户间交互和协作

**优点**：
- 满足隐性需求中的社交需求
- 用户活跃度高，用户间互助
- 社区文化形成，用户归属感强
- 社区内容自生长，降低运营成本

**缺点**：
- 开发成本高，需要社交功能架构
- 用户管理复杂，需要治理机制
- 社区活跃度培养困难
- 风险控制（如负面内容）难度大

**适用场景**：
- 用户群体同质性强
- 社交需求明确
- 技术团队规模大

### 决策理由

我们选择**陪伴型助手 + 社区型平台混合模式**，具体决策如下：

#### 核心服务模式：陪伴型助手
**满足需求**：
- 显性需求：座位查找、图书检索等功能服务
- 隐性需求：个性化推荐、学习陪伴、情感支持

**具体实现**：
1. **智能推荐助手**
   - 根据用户身份（考研学生、留学生、教师、新生）提供个性化推荐
   - 推荐座位、书籍、资源、活动
   - 提供学习路径规划建议

2. **学习陪伴助手**
   - 学习时间记录和分析
   - 学习效率提醒
   - 学习目标设定和追踪
   - 情感支持提示（如疲劳提醒、鼓励语句）

3. **情境感知助手**
   - 根据用户当前情境（如考试周、假期）调整推荐策略
   - 根据图书馆实时状态（如座位紧张度）调整推荐
   - 根据用户历史行为优化服务

#### 补充服务模式：社区型平台（轻量级）
**满足需求**：
- 隐性需求中的社交需求（考研社群连接、新生社交融入、学术合作匹配）

**具体实现**：
1. **学习小组匹配**
   - 根据用户学习目标和兴趣匹配学习小组
   - 提供小组协作工具（如共享笔记、讨论区）
   - 小组学习记录和成果展示

2. **社群连接功能**
   - 考研学生社群：分享备考经验、心理支持
   - 留学生社群：跨文化交流、学习伙伴匹配
   - 教师社群：学术合作匹配、研究资源共享

3. **UGC内容平台**
   - 用户分享学习心得、笔记、书评
   - 用户推荐书籍和资源
   - 用户互相问答（轻量级问答系统）

### 决策依据

1. **需求匹配度**：
   - 第一阶段调研显示，用户隐性需求占比高（26项隐性需求 vs 10项显性需求）
   - 考研学生、留学生、新生等群体有明确的社交和情感支持需求
   - 纯工具型系统无法满足这些需求

2. **用户价值最大化**：
   - 陪伴型助手能建立用户情感连接，提升用户粘性
   - 社区型平台能解决用户社交需求，提升用户活跃度
   - 混合模式能最大化用户价值

3. **技术可行性**：
   - 陪伴型助手：现有AI技术（推荐算法、用户画像）可支撑
   - 社区型平台（轻量级）：现有社交功能架构可支撑
   - 技术团队有能力实现混合模式

4. **差异化竞争**：
   - 竞品多为工具型系统，陪伴型+社区型可作为差异化竞争点
   - 用户情感连接和社群归属感是难以复制的竞争力

5. **资源约束平衡**：
   - 纯社区型平台开发成本过高，团队资源有限
   - 轻量级社区功能（如学习小组匹配）可在资源约束下实现

### 决策后果

#### 正面后果
1. 用户满意度提升：不仅满足功能需求，还满足情感和社交需求
2. 用户粘性提升：陪伴型助手建立情感连接，社区型平台建立归属感
3. 差异化竞争优势：独特的陪伴+社区模式
4. 用户价值最大化：从工具升级为伙伴和社区

#### 负面后果
1. 开发成本增加：需要开发AI推荐系统和社交功能
2. 技术架构复杂：需要集成推荐算法、用户画像、社交功能
3. 运营成本增加：需要持续优化推荐算法、管理社区内容
4. 风险增加：社区内容治理风险、用户隐私保护风险

#### 风险缓解措施
1. 分阶段实施：先上线陪伴型助手核心功能，后续迭代社区功能
2. 技术选型：采用成熟的开源框架降低开发成本
3. 社区治理：建立内容审核机制和用户行为规范
4. 隐私保护：严格遵守数据保护法规，用户数据脱敏处理

## 技术架构概览

### 系统分层架构

```
┌─────────────────────────────────────────┐
│          用户交互层                      │
│  Web端 | 移动端 | 智能助手接口           │
└─────────────────────────────────────────┘
┌─────────────────────────────────────────┐
│          服务层                          │
│  功能服务 | 推荐服务 | 社区服务           │
└─────────────────────────────────────────┘
┌─────────────────────────────────────────┐
│          智能层                          │
│  推荐算法 | 用户画像 | 情境感知           │
└─────────────────────────────────────────┘
┌─────────────────────────────────────────┐
│          数据层                          │
│  用户数据 | 图书数据 | 行为数据           │
└─────────────────────────────────────────┘
```

### 核心模块

1. **功能服务模块**：座位查找、图书检索、借还管理
2. **推荐服务模块**：个性化推荐、情境感知推荐
3. **社区服务模块**：学习小组、社群连接、UGC平台
4. **智能助手模块**：学习陪伴、情感支持、问答系统

## 验证标准

### 用户满意度指标
- 功能满意度：显性需求满足率≥80%
- 陪伴满意度：隐性需求满足率≥60%
- 社区满意度：社交需求满足率≥50%

### 用户粘性指标
- 日活跃用户（DAU）增长≥20%
- 用户平均使用时长增长≥30%
- 用户留存率（30天）≥40%

### 用户价值指标
- 用户推荐书籍阅读率≥30%
- 学习小组匹配成功率≥40%
- 用户UGC内容贡献率≥10%
